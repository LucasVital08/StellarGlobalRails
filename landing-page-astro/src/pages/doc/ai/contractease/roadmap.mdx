## Phase 2 — Real Escrow with Claimable Balances

**Target:** Q3 2025

The escrow feature stub exists in the codebase as a `createEscrow()` call that currently throws `NotImplementedError`. Phase 2 replaces this with a real Stellar Claimable Balance implementation.

**Scope:**

- Buyer links a contract to an escrow amount in XLM or USDC
- Clicking "Fund Escrow" opens Freighter → user signs a `createClaimableBalance` operation
- The balance ID is stored in a new `escrows` table linked to `contracts.id`
- Seller can claim after the contract `completed` date + configured release delay
- Buyer can trigger refund if contract is `cancelled` before delivery
- UI: escrow status widget on `ContractDetail` page

**New table:**

```sql
create table escrows (
  id             uuid primary key default gen_random_uuid(),
  contract_id    uuid not null references contracts(id),
  organization_id uuid not null references organizations(id),
  balance_id     text unique not null,   -- Stellar Claimable Balance ID
  asset_code     text not null,          -- 'XLM' | 'USDC'
  amount         decimal not null,
  status         text not null default 'locked',
    -- 'locked' | 'claimed' | 'refunded' | 'expired'
  funded_by      text not null,          -- buyer Stellar address
  claimant       text not null,          -- seller Stellar address
  release_at     timestamptz not null,
  created_at     timestamptz not null default now()
);
```

## Phase 3 — x402 API Monetization + Soroban Smart Clauses

**Target:** Q4 2025

### x402 Integration

The x402 protocol allows HTTP APIs to require micropayments per request. ContractEase will use x402 to monetize its API for third-party integrators:

- `POST /api/contracts/anchor` — pay per anchor operation (0.10 USDC)
- `GET /api/contracts/:id/verify` — free read, paid verification certificate PDF
- Kivo Pay handles the x402 payment channel (SGR ecosystem integration)

### Soroban Smart Clauses

Soroban is Stellar's native smart contract platform. Phase 3 introduces **smart clauses** — contract conditions that execute automatically on Stellar when triggered:

| Clause Type | Trigger | Soroban Action |
|---|---|---|
| Auto-release escrow | All parties signed + N days elapsed | `claim_balance()` |
| Penalty clause | Milestone missed | Transfer penalty amount to harmed party |
| Expiry clause | Contract date passed | Mark invalid on-chain |

**Technical approach:**
- Smart contracts written in Rust, deployed to Soroban Mainnet
- `contract_clauses.clause_type = 'soroban'` stores the contract address and invocation parameters
- Edge Function `invoke-soroban-clause` triggers the contract using the Soroban client SDK

## Phase 4 — Stellar Mainnet + ICP-Brasil Compliance

**Target:** Q1 2026

### Stellar Mainnet

- Switch `VITE_STELLAR_NETWORK=mainnet` in production
- Fund custodial keypair on Mainnet (XLM purchase)
- Update Horizon URL to `https://horizon.stellar.org`
- Verify all Explorer links point to Mainnet
- Performance test: confirm 3–5 s finality under load

### ICP-Brasil Compliance

ICP-Brasil (Infraestrutura de Chaves Públicas Brasileira) is the Brazilian government's PKI standard for legally valid digital signatures. Integration enables ContractEase contracts to have the same legal weight as notarized documents.

**Implementation options under evaluation:**
1. **Certisign API** — Brazilian CA, issues ICP-Brasil certificates via REST API
2. **Docusign + ICP-Brasil module** — existing integration, higher cost
3. **BirdID SDK** — mobile-based ICP-Brasil certificate, most user-friendly

**What changes:**
- Signing flow adds an optional "Use ICP-Brasil certificate" step
- Certificate data is stored in `contract_parties.certificate_data`
- PDF includes the ICP-Brasil signature block (PKCS#7 format)
- Stellar anchor still provides blockchain proof-of-existence (complementary, not replacement)

### e-CNPJ Integration

For organizations signing contracts as legal entities (not individuals), e-CNPJ certificates provide company-level authentication. Integration mirrors the e-CPF/ICP-Brasil flow.

## Phase 5 — Mobile App + Offline Signing

**Target:** Q2 2026

### React Native App

- Shared business logic with the web app (hooks, types, Supabase client)
- Expo managed workflow for iOS and Android
- Freighter Mobile wallet integration (when available) or WalletConnect
- Push notifications for signing requests (Expo Push Notifications)
- Biometric authentication for app access

### Offline Signing

Brazilian infrastructure can be unreliable. Offline signing allows parties to sign contracts without an internet connection:

- Contract document cached locally (IndexedDB / AsyncStorage)
- Signature captured offline
- Sync queue: signature uploaded when connection resumes
- OTP validation deferred: offline-signed contracts marked `pending_verification` until online

## Open Technical Decisions

| Decision | Options Under Consideration | Status |
|---|---|---|
| PDF signing standard | PDF/A-3 (standard), PKCS#7 (ICP-Brasil) | Evaluating ICP-Brasil SDK cost |
| Email provider | Resend (current), SendGrid, AWS SES | Resend performing well; revisit at scale |
| Rate limiting | Upstash Redis (edge), Supabase pg_cron cleanup | Evaluating Upstash for OTP abuse prevention |
| Contract storage encryption | AES-256 at rest (Supabase default), client-side encryption | Client-side adds complexity; evaluating demand |
| Template marketplace | Public templates shareable between orgs | Architecture designed; pending product decision |
| Multi-language contracts | Side-by-side pt-BR / en clauses | Requested by export-focused customers |
| Webhook support | Outbound webhooks for contract events | High demand from integration partners |

## Contribution Guide

ContractEase follows the SGR monorepo conventions:

```bash
# Feature branches
git checkout -b feat/escrow-claimable-balances

# Commit format (Conventional Commits)
git commit -m "feat(escrow): implement createClaimableBalance Edge Function"
git commit -m "fix(signing): handle expired sign_token gracefully"
git commit -m "docs(roadmap): add Phase 3 Soroban details"

# Before PR
npm run lint
npm run type-check
npm run test
```

All PRs require:
- TypeScript `strict: true` compliance
- No new `any` types without comment justification
- RLS policy added for any new table
- Edge Function tested with `supabase functions serve`
