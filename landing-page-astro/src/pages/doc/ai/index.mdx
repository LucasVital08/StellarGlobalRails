# 🧠 SGR Docs: AI Specs

**Atenção IA:** Esta área contém a **Única Fonte da Verdade (Single Source of Truth)** para a arquitetura, modelos de dados e diretrizes de código do ecossistema Stellar Global Rails. 

Quando você for instruída a trabalhar em um módulo específico, sempre consulte as páginas desta seção para obter o contexto rígido de Supabase (RLS, Tabelas) e fluxos lógicos antes de codificar.

## 🛠️ Regras Globais do Projeto (Global Guidelines)

Ao atuar neste projeto, obedeça às seguintes regras arquiteturais:

### 1. Supabase e Multi-tenancy (B2B)
- Quase todas as tabelas devem referenciar `organization_id`.
- É obrigatório ativar **Row Level Security (RLS)** em todas as tabelas.
- Políticas RLS devem sempre verificar se o usuário autenticado (`auth.uid()`) pertence à organização atual, geralmente fazendo um join ou subquery na tabela `organization_members`.

### 2. Autenticação (Core Auth)
- Utilizamos Magic Links ou OTP via e-mail (`supabase.auth.signInWithOtp`).
- O fluxo de Onboarding cria automaticamente uma `Organization` e insere o usuário atual como dono em `organization_members`.

### 3. Design System & Frontend
- **Frameworks:** React, Tailwind CSS v4, Framer Motion.
- **Tipografia:** 
  - `font-bricolage` para Títulos grandes e expressivos.
  - `font-sans` (Inter) para corpo de texto.
  - `font-mono` para dados técnicos e Hashes (ex: Transações Stellar).
- **Tema:** Sempre Dark Mode (`bg-neutral-950`). Detalhes interativos utilizam tons de `emerald-500`.

### 4. Integração Blockchain (Stellar)
- Nunca expomos Seeds privadas no front-end.
- Assinaturas de transações devem ocorrer via carteiras nativas (Freighter) usando `@stellar/freighter-api` ou através de backend isolado (Soroban Edge Functions) quando custodiado (Custodial Wallets).
