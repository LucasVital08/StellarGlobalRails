# 🗄️ Supabase & RLS Guidelines

A segurança do SGR baseia-se no princípio de isolamento total entre organizações via **Row Level Security (RLS)**.

---

## Arquitetura Multi-tenant

O campo `organization_id` (UUID) é a chave de isolamento em 90% das tabelas.

### Verificação de Pertencimento
Para garantir que um usuário só acesse dados de sua org, usamos a tabela `organization_members`.

**Política SQL Exemplo (RLS):**
```sql
-- Política para a tabela 'contracts'
CREATE POLICY "Users can view contracts from their organization" 
ON public.contracts
FOR SELECT
USING (
  organization_id IN (
    SELECT organization_id 
    FROM public.organization_members 
    WHERE user_id = auth.uid()
  )
);
```

---

## Auth Flow

- **Tipo:** Passwordless (Magic Link / OTP).
- **Session:** JWT gerenciado pelo Supabase Client.
- **Hook de Contexto:** `useAuth` e `useOrganization`.

---

## Edge Functions (Soroban/Stellar)

Operações que exigem chaves privadas (custodial) ou interação direta com a blockchain ocorrem em Edge Functions para garantir que o segredo nunca vaze para o cliente.

- **Local:** `supabase/functions/`
- **Linguagem:** TypeScript (Deno).

---

## Regras para a IA
1. **Sempre ative RLS:** Nunca crie uma tabela sem políticas.
2. **Subqueries Eficientes:** Ao criar políticas RLS, use subqueries indexadas para evitar gargalos de performance.
3. **Database Types:** Sempre exporte e use os tipos gerados pelo Supabase CLI (`types/supabase.ts`).
