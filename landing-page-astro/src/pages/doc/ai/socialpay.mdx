# 💬 Specs: SocialPay

> **Para IA:** Este módulo ainda está em fase de planejamento/MVP. As tabelas abaixo são a especificação de referência para quando o desenvolvimento iniciar.

---

## Visão Geral

O SocialPay permite pagamentos corporativos internos via `@handles`. Cada colaborador ou departamento recebe um handle legível, e transferências são feitas sem expor dados bancários.

**Status:** Simulador funcional na landing page. Backend ainda não implementado.

---

## Modelo de Dados (Planejado)

### `social_handles`
Mapeamento de handles para carteiras Stellar.

| Coluna | Tipo | Notas |
|---|---|---|
| `id` | `uuid` (PK) | Auto-gerado |
| `organization_id` | `uuid` | FK → `organizations.id` |
| `handle` | `text` | Ex: `@marketing`, `@joao.silva` — UNIQUE por org |
| `display_name` | `text` | Nome de exibição |
| `stellar_public_key` | `text` | Chave pública Stellar associada |
| `type` | `text` | `'user'` \| `'department'` \| `'external'` |
| `is_active` | `bool` | Default `true` |
| `created_at` | `timestamptz` | Timestamp |

### `social_transactions`
Feed auditável de pagamentos internos.

| Coluna | Tipo | Notas |
|---|---|---|
| `id` | `uuid` (PK) | Auto-gerado |
| `organization_id` | `uuid` | FK → `organizations.id` |
| `from_handle_id` | `uuid` | FK → `social_handles.id` |
| `to_handle_id` | `uuid` | FK → `social_handles.id` |
| `amount` | `numeric` | Valor em USDC |
| `memo` | `text?` | Descrição do pagamento |
| `stellar_tx_hash` | `text?` | Hash da transação Stellar |
| `status` | `text` | `'pending'` \| `'confirmed'` \| `'failed'` |
| `created_at` | `timestamptz` | Timestamp |

### RLS Obrigatório
- Ambas as tabelas filtram por `organization_id`.
- Apenas membros da organização (via `organization_members`) podem ver/criar transações.
- Admins têm acesso total ao feed; membros veem apenas transações onde são parte.

---

## Fluxo de Pagamento

```
1. Remetente digita @handle-destino + valor + memo (opcional)
2. Frontend resolve handle → stellar_public_key (via social_handles)
3. Backend cria transação na Stellar via Kivo Pay
4. Transação confirmada → insere em social_transactions
5. Feed atualiza em tempo real via SSE (Server-Sent Events)
6. Destinatário recebe notificação in-app
```

## Feed em Tempo Real

O feed auditável deve usar **SSE (Server-Sent Events)** ou **Supabase Realtime** para atualizar automaticamente quando uma nova transação for inserida.

```typescript
// Exemplo com Supabase Realtime
supabase
  .channel('social-feed')
  .on('postgres_changes', {
    event: 'INSERT',
    schema: 'public',
    table: 'social_transactions',
    filter: `organization_id=eq.${currentOrgId}`
  }, (payload) => {
    addToFeed(payload.new);
  })
  .subscribe();
```

## Regras para a IA

1. **Handles são únicos por organização**, não globalmente.
2. **Sempre filtrar por `organization_id`** em todas as queries.
3. O simulador na landing page (`KivoSocialPaySimulator.tsx`) é apenas visual — o backend real seguirá esta spec.
4. Usar `supabase.rpc()` para resolver handles, evitando expor a tabela diretamente ao frontend.
