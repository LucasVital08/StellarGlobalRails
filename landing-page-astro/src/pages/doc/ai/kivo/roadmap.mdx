## Phase 1 — Foundation (Now)

**Goal:** Working Go API with device registration, basic payments, and Supabase schema running on Fly.io testnet.

**Deliverables:**

- Go + Fiber project scaffold with `sqlc`, `pgx v5`, `go-redis`
- Supabase schema migrations applied (devices, payments, webhooks, api_keys)
- `POST /v1/devices` — register device, generate Stellar keypair, issue API key
- `GET /v1/devices/:id` — fetch device + Stellar balance via Horizon
- `POST /v1/payments` — create payment (no condition, immediate Stellar submission)
- `GET /v1/payments/:id` — payment status
- `GET /v1/health` — health check with DB + Redis + Horizon ping
- Device API key middleware (SHA-256 hash lookup)
- Supabase Auth JWT middleware
- Payment worker goroutine (Redis → Stellar submission)
- Testnet Friendbot funding helper for dev
- Fly.io deployment with secrets
- GitHub Actions CI: `go test ./...`, `sqlc generate`, Docker build

**Timeline:** 3–4 weeks

## Open Technical Decisions

### 1. RabbitMQ vs Redis for Queue at Scale

**Current:** Redis (Upstash) with `LPUSH`/`BRPOP`.

**Problem at scale:** Redis lists provide no message acknowledgment. If a payment worker crashes after dequeuing but before submitting to Stellar, the payment is lost.

**Options:**
- **Redis Streams** (`XADD`/`XREADGROUP`): adds consumer groups and acknowledgment without a new dependency. Simple upgrade path.
- **RabbitMQ** (self-hosted or CloudAMQP): full message broker with dead letter queues, routing, and durable acknowledgment. Better for complex routing (payment events → webhook worker → retry queue).
- **NATS JetStream**: high-throughput, lightweight, great for M2M messaging. Slightly less ecosystem support in Go compared to RabbitMQ.

**Decision criteria:** Migrate when payment volume exceeds 5,000 payments/day or when a payment loss incident occurs in production.

**Current lean:** Redis Streams first (zero new dependencies), then RabbitMQ if routing complexity grows.

---

### 2. Rust Rewrite of Payment Hot Path

**Context:** The payment submission path (decrypt Stellar secret → build transaction → sign → submit to Horizon) is latency-sensitive. For EV charging or energy trading use cases, every millisecond of settlement delay has user-visible impact.

**Options:**
- **Stay in Go:** Already fast (sub-millisecond signing). Horizon network latency (200–500ms) dominates — Go vs Rust makes no difference.
- **Rust sidecar for signing:** The secret decryption and transaction signing runs in a Rust binary called via Unix socket. Go handles HTTP, Rust handles crypto. Reduces attack surface for secret handling.
- **Full Rust rewrite:** Axum + sqlx. Maximum performance, but significant rewrite cost.

**Current decision:** Stay in Go. Revisit at Phase 4 when Soroban contract invocation adds complexity.

---

### 3. Soroban vs Claimable Balances for Conditional Payments

**Current:** Claimable Balances for escrow-style conditional payments (lock funds, release on condition, refund on timeout).

**Problem:** Claimable Balances are simple but not programmable. The condition logic (energy_kwh threshold, oracle verification) runs on Kivo's server, not on-chain. This requires trust in Kivo.

**Soroban option:** Deploy a Soroban contract that holds the escrow, verifies the condition (via signed oracle data submitted to the contract), and releases funds trustlessly. No Kivo server involvement after payment creation.

**Trade-offs:**

| | Claimable Balances | Soroban |
|---|---|---|
| Trustless | No (Kivo verifies) | Yes (contract verifies) |
| Gas cost | ~0.00001 XLM | 0.01–0.10 XLM |
| Complexity | Low | High |
| Latency | 3–5s | 3–5s |
| Available now | Yes | Testnet only |

**Current decision:** Claimable Balances for Phase 1–3. Soroban for Phase 4+ where trustless verification is required (energy trading, AI agent economy).

---

### 4. Self-Hosted Supabase vs Managed at Scale

**Current:** Supabase managed (free tier → $25/month plan).

**Problem:** At 1 million+ registered devices, the Supabase managed plan becomes expensive (~$600+/month for the Pro plan with add-ons). Each device creates rows in `devices`, potentially multiple rows in `payments` per day.

**Options:**
- **Supabase Pro ($25/month):** 8 GB storage, 15 GB bandwidth — handles ~800k devices.
- **Supabase Team ($599/month):** For serious production scale.
- **Self-hosted Supabase on Fly.io:** Run `supabase/postgres` Docker image on Fly.io volumes. ~$20/month for 50 GB. Operational burden increases significantly (backups, HA, upgrades).
- **Neon Serverless Postgres:** Branching for dev/prod, autoscaling — different from Supabase; loses RLS integration.

**Current decision:** Managed Supabase through $25/month plan. Evaluate self-hosted at 500k+ devices.

## Changelog

### 2025-05-15 — Documentation Complete
All 10 documentation files written covering architecture, data models, API reference, auth, Stellar integration, x402, MCP, deployment, and roadmap.

### 2025-04-28 — ADR-006: x402 Protocol Adopted
Decided to implement x402 as the machine payment protocol rather than a custom header scheme. Aligned with emerging open standard for HTTP micropayments.

### 2025-04-15 — ADR-005: Upstash Redis Selected
Selected Upstash Redis over self-hosted Redis for the job queue. Free tier sufficient for MVP. go-redis v9 client chosen for its context-aware API.

### 2025-04-01 — ADR-001–004: Core Stack Finalized
Go + Fiber, sqlc, pgx v5, and Supabase PostgreSQL selected as the core technology stack. Decision documented in ADRs to prevent future revisiting.

### 2025-03-20 — MCP Server Added to Scope
After reviewing the Anthropic MCP specification, the Kivo MCP server was added to the product roadmap as Layer 3. Tool specifications drafted for `kivo_create_payment`, `kivo_check_status`, `kivo_execute_contract`, `kivo_subscribe_webhook`, `kivo_simulate_payment`.

### 2025-03-01 — Project Initiated
Kivo Pay initiated as the third product in the Stellar Global Rails platform, targeting M2M payments and the emerging AI Agent Economy.
