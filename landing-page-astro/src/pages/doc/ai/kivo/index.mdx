# Kivo Pay — Product Overview

> **Machines need a network. Kivo is the network.**

Kivo Pay is an M2M (machine-to-machine) payment SDK that gives autonomous devices, IoT sensors, AI agents, and embedded systems the ability to transact with each other — without human intervention, without bank accounts, and without KYC.

---

## Why M2M Payments Are Hard Today

Modern infrastructure is filled with machines that need to exchange value: EV chargers billing cars, smart meters settling with energy grids, AI agents paying for API access, drones paying landing fees. Yet every existing payment rail was designed for humans.

| Problem | Why it blocks machines |
|---|---|
| Bank accounts require identity | Machines cannot complete KYC/AML processes |
| IBAN/SWIFT require legal entities | A Raspberry Pi cannot be a legal person |
| Card networks require a human cardholder | No machine can hold a Visa card |
| ACH/SEPA settle in days | A charging session needs settlement in seconds |
| OAuth flows require browser redirects | An embedded device has no browser |
| Traditional APIs use API keys + billing | No machine-native micropayment primitive |

The result: developers building M2M payment flows today must either (a) route through a human-controlled account, creating a bottleneck and liability, or (b) build complex off-chain accounting that introduces trust and reconciliation problems.

---

## How Stellar Solves This

Stellar was designed as a global payment network, and it happens to be a near-perfect substrate for M2M payments:

| Stellar Property | M2M Benefit |
|---|---|
| Wallet per keypair (no KYC) | Any device gets a wallet in milliseconds |
| $0.00001 per transaction | Micropayments are economically viable |
| 3–5 second settlement | Real-time device interactions |
| USDC / XLM native assets | Stable-value payments without volatility risk |
| Claimable Balances | Escrow-style conditional payments |
| Path Payments / DEX | Cross-asset payments automatically routed |
| Soroban smart contracts | Complex programmable payment logic |
| Open source, no permission | No vendor lock-in for device manufacturers |

A device gets a Stellar keypair. Its public key is its wallet address. It can send and receive payments, hold balances, and participate in the network without any human in the loop.

## Three Product Layers

Kivo Pay is not a single library — it is a three-layer stack, each layer building on the one below.

```
┌─────────────────────────────────────────────────────┐
│  Layer 3 — MCP Server                               │
│  AI agents (Claude, GPT-4) call Kivo tools natively │
│  Standard Model Context Protocol interface          │
└─────────────────────────────┬───────────────────────┘
                              │
┌─────────────────────────────▼───────────────────────┐
│  Layer 2 — Programmable Gateway (REST API)          │
│  Go + Fiber HTTP API                                │
│  x402 middleware, webhooks, conditional payments    │
└─────────────────────────────┬───────────────────────┘
                              │
┌─────────────────────────────▼───────────────────────┐
│  Layer 1 — M2M SDK                                  │
│  Developer library for device integration           │
│  TypeScript (browser/Node) + Go client              │
└─────────────────────────────────────────────────────┘
```

### Layer 1: M2M SDK

The developer-facing library that device firmware or edge software embeds. It handles wallet generation, payment signing, condition evaluation, and communication with the Gateway. Available as a TypeScript package (`@kivo/sdk`) and a Go module (`github.com/kivo-pay/sdk`).

### Layer 2: Programmable Gateway

The Go + Fiber HTTP API that orchestrates payments. It registers devices, manages API keys, stores payment records in Supabase PostgreSQL, submits transactions to Stellar Horizon, processes async jobs via Redis, and fires webhooks on payment events. This is the core of Kivo Pay.

### Layer 3: MCP Server

A Model Context Protocol server that exposes Kivo tools to AI agents. Claude, GPT-4, and any MCP-compatible agent can call `kivo_create_payment`, `kivo_execute_contract`, and `kivo_check_status` natively, without any custom integration. This is the gateway to the AI Agent Economy.

## x402 Protocol

The x402 protocol repurposes the HTTP 402 Payment Required status code as a machine-native payment primitive. When a device requests a protected resource, the server responds with `402` and a `X-PAYMENT-REQUIRED` header describing what to pay. The device signs a Stellar payment, encodes it in an `X-PAYMENT` header, and retries the request. The server verifies and settles.

This requires no OAuth, no redirect, no session — just HTTP headers. Any HTTP client, from a Python script to a C firmware library, can implement x402.

## Use Case Overview

| Use Case | Devices Involved | Payment Flow | Asset |
|---|---|---|---|
| EV Charging | Car ↔ Charger | Car pays per kWh as energy flows | USDC |
| P2P Energy Trading | Solar Panel ↔ Grid | Panel sells excess energy in real time | USDC |
| AI Agent Economy | Agent ↔ API Provider | Agent pays per API call via x402 | USDC |
| Drone Logistics | Drone ↔ Landing Pad | Drone pays landing fee autonomously | XLM |
| IoT Data Marketplace | Sensor ↔ Data Buyer | Sensor streams data, buyer pays per record | USDC |
| Edge Compute | Client ↔ GPU Node | Client pays per compute unit | USDC |

### EV Charging

An EV connects to a charger. The charger's firmware runs the Kivo SDK. A payment is created with a condition: `energy_kwh = 10`. As the car draws energy, the kWh counter increments. When the condition is met, the Stellar transaction is submitted automatically. The car's wallet is debited; the charger's wallet is credited. No human touched the transaction.

### P2P Energy Trading

A rooftop solar panel is generating more than its household needs. The Kivo SDK on its edge controller creates a payment channel with the local grid. As excess watts flow, micropaymentssettle in real time on Stellar. The panel owner earns USDC per kWh, not once a month but second by second.

### AI Agent Economy

An AI agent (Claude, running autonomously) needs to call a premium weather data API. The API is protected by x402. The agent uses the `kivo_create_payment` MCP tool to authorize a payment, encodes it in the request header, and receives the data. The entire flow — payment negotiation, signing, and verification — happens inside a single agent tool call.

## How Kivo Fits in the SGR Platform

Stellar Global Rails (SGR) is a platform of three complementary products. All three share infrastructure while serving distinct use cases:

| Product | Layer | Stellar Role |
|---|---|---|
| ContractEase | Smart contracts | Deploy + interact with Soroban contracts |
| SocialPay | Consumer payments | Human-to-human Stellar transfers |
| Kivo Pay | M2M payments | Device-to-device autonomous payments |

All three products share the same Supabase PostgreSQL cluster. Kivo reuses the Supabase Auth system for dashboard users (device owners) while extending it with device API keys for M2M authentication. The same Stellar network (testnet for dev, mainnet for production) underlies all three products, which means Kivo devices can pay ContractEase-deployed contracts and SocialPay users in the same transaction graph.
