# Deployment Guide

This guide covers the full deployment of SocialPay after the Supabase refactor, including the database migration from LibSQL, environment variable configuration, local development setup, and production deployment options.

---

## Migration: LibSQL → Supabase

This section documents the steps to migrate an existing SocialPay instance from Prisma + LibSQL to Supabase PostgreSQL.

### Step 1: Create the Supabase Project

1. Go to [supabase.com](https://supabase.com) and create a new project
2. Choose a region close to your users (e.g., `sa-east-1` for Brazil)
3. Save the database password — you will need it for direct connections
4. Wait for provisioning (~2 minutes)

From the project settings, collect:
- `Project URL` → `NEXT_PUBLIC_SUPABASE_URL`
- `Anon/public key` → `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `Service role key` → `SUPABASE_SERVICE_ROLE_KEY`
- `JWT secret` → `SUPABASE_JWT_SECRET`

### Step 2: Run SQL Migrations

Install the Supabase CLI and run the migrations:

```bash
npm install -g supabase
supabase login
supabase link --project-ref YOUR_PROJECT_REF
supabase db push
```

Or run the SQL directly in the Supabase SQL editor. The full migration is in `supabase/migrations/001_initial.sql`:

```sql
-- Enable necessary extensions
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- Organizations
CREATE TABLE organizations (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name       text NOT NULL,
  slug       text NOT NULL UNIQUE,
  owner_id   uuid REFERENCES auth.users(id),
  plan       text NOT NULL DEFAULT 'free',
  created_at timestamptz NOT NULL DEFAULT now()
);
ALTER TABLE organizations ENABLE ROW LEVEL SECURITY;

-- Profiles (extends auth.users)
CREATE TABLE profiles (
  id                 uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  handle             text,
  display_name       text,
  avatar_url         text,
  stellar_public_key text UNIQUE,
  organization_id    uuid REFERENCES organizations(id),
  handle_type        text NOT NULL DEFAULT 'user',
  created_at         timestamptz NOT NULL DEFAULT now(),
  UNIQUE (handle, organization_id)
);
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

-- Trigger: create profile on signup
CREATE OR REPLACE FUNCTION handle_new_user() RETURNS trigger AS $$
BEGIN
  INSERT INTO profiles (id) VALUES (NEW.id);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION handle_new_user();

-- Organization members
CREATE TABLE organization_members (
  id        uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id    uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  user_id   uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role      text NOT NULL DEFAULT 'member',
  joined_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (org_id, user_id)
);
ALTER TABLE organization_members ENABLE ROW LEVEL SECURITY;

-- Wallets
CREATE TABLE wallets (
  id                 uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id            uuid NOT NULL UNIQUE REFERENCES auth.users(id) ON DELETE CASCADE,
  stellar_public_key text NOT NULL UNIQUE,
  vault_secret_id    text NOT NULL,
  created_at         timestamptz NOT NULL DEFAULT now()
);
ALTER TABLE wallets ENABLE ROW LEVEL SECURITY;

-- Transactions
CREATE TABLE transactions (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  sender_id       uuid NOT NULL REFERENCES auth.users(id),
  receiver_id     uuid NOT NULL REFERENCES auth.users(id),
  sender_handle   text NOT NULL,
  receiver_handle text NOT NULL,
  amount          numeric(18, 7) NOT NULL,
  asset_code      text NOT NULL DEFAULT 'XLM',
  description     text,
  visibility      text NOT NULL DEFAULT 'org',
  status          text NOT NULL DEFAULT 'pending',
  stellar_hash    text UNIQUE,
  explorer_url    text,
  organization_id uuid REFERENCES organizations(id),
  created_at      timestamptz NOT NULL DEFAULT now(),
  confirmed_at    timestamptz
);
ALTER TABLE transactions ENABLE ROW LEVEL SECURITY;

-- Ramp orders
CREATE TABLE ramp_orders (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id        uuid NOT NULL REFERENCES auth.users(id),
  type           text NOT NULL,
  fiat_amount    numeric(18, 2) NOT NULL,
  fiat_currency  text NOT NULL,
  crypto_asset   text NOT NULL DEFAULT 'USDC',
  crypto_amount  numeric(18, 7),
  status         text NOT NULL DEFAULT 'pending',
  etherfuse_ref  text UNIQUE,
  stellar_hash   text,
  payment_url    text,
  expires_at     timestamptz,
  created_at     timestamptz NOT NULL DEFAULT now(),
  completed_at   timestamptz
);
ALTER TABLE ramp_orders ENABLE ROW LEVEL SECURITY;

-- Indexes
CREATE INDEX idx_profiles_handle_org ON profiles (handle, organization_id);
CREATE INDEX idx_transactions_org_vis_created ON transactions (organization_id, visibility, created_at DESC);
CREATE INDEX idx_transactions_sender ON transactions (sender_id, created_at DESC);
CREATE INDEX idx_transactions_receiver ON transactions (receiver_id, created_at DESC);
```

### Step 4: Update Environment Variables

Remove all LibSQL/Prisma variables and add Supabase variables:

```bash
# Remove (LibSQL era)
DATABASE_URL=
NEXTAUTH_SECRET=
NEXTAUTH_URL=

# Add (Supabase era)
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...
SUPABASE_SERVICE_ROLE_KEY=eyJ...
SUPABASE_JWT_SECRET=your-jwt-secret
```

### Step 5: Replace Prisma Client

Replace all Prisma imports with Supabase client calls:

```typescript
// BEFORE (Prisma)
import { prisma } from '@/lib/prisma';
const user = await prisma.user.findUnique({ where: { email } });

// AFTER (Supabase)
import { createClient } from '@/lib/supabase-server';
const supabase = createClient();
const { data: user } = await supabase
  .from('profiles')
  .select('*')
  .eq('id', userId)
  .single();
```

### Step 6: Migrate Auth

Remove custom session middleware and replace with Supabase middleware:

```bash
# Remove
rm lib/auth.ts lib/session.ts middleware/session.ts

# Update
# - middleware.ts → use Supabase session check (see auth-security.mdx)
# - app/api/auth/login → replace with POST /api/auth/magic-link
# - app/api/auth/logout → replace with supabase.auth.signOut()
```

## Environment Variables Reference

| Variable | Required | Description |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | Yes | Supabase project URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Yes | Supabase anon/public key (safe for browser) |
| `SUPABASE_SERVICE_ROLE_KEY` | Yes | Service role key (server only — never expose to browser) |
| `SUPABASE_JWT_SECRET` | Yes | For custom JWT verification |
| `STELLAR_NETWORK` | Yes | `testnet` or `mainnet` |
| `ETHERFUSE_API_KEY` | Yes | Etherfuse API key |
| `ETHERFUSE_WEBHOOK_SECRET` | Yes | HMAC secret for webhook signature verification |
| `PLATFORM_PUBLIC_KEY` | Yes | Platform's Stellar public key (x402 payment destination) |
| `UPSTASH_REDIS_REST_URL` | Optional | For rate limiting (recommended in production) |
| `UPSTASH_REDIS_REST_TOKEN` | Optional | Upstash Redis token |
| `NEXT_PUBLIC_APP_URL` | Yes | App base URL (for magic link redirect) |

## Local Development

```bash
# 1. Install dependencies
npm install

# 2. Start local Supabase (Docker required)
npx supabase start
# Output: API URL, anon key, service role key for local use

# 3. Copy env template
cp .env.example .env.local
# Fill in the local Supabase values from step 2

# 4. Apply migrations to local Supabase
npx supabase db reset   # applies all migrations fresh

# 5. Start Next.js
npm run dev
# App running at http://localhost:3000
# Supabase Studio at http://localhost:54323
```

## Deploy: Vercel (Recommended)

Vercel has first-class Next.js support and zero-config deployment.

```bash
# Install Vercel CLI
npm install -g vercel

# Deploy
vercel deploy --prod

# Set environment variables in Vercel dashboard
# or via CLI:
vercel env add NEXT_PUBLIC_SUPABASE_URL production
vercel env add SUPABASE_SERVICE_ROLE_KEY production
# ... (add all variables from the table above)
```

Connect the Vercel project to your GitHub repo for automatic deployments on push to `main`.

## Deploy: Cloudflare Pages

Cloudflare Pages requires a Next.js adapter for full compatibility.

```bash
npm install @cloudflare/next-on-pages
```

```javascript
// next.config.js
const { setupDevPlatform } = require('@cloudflare/next-on-pages/next-dev');

/** @type {import('next').NextConfig} */
const nextConfig = {
  // Required for Cloudflare Pages
};

if (process.env.NODE_ENV === 'development') {
  setupDevPlatform();
}

module.exports = nextConfig;
```

```toml
# wrangler.toml
name = "socialpay"
compatibility_date = "2024-01-01"
compatibility_flags = ["nodejs_compat"]

[vars]
NEXT_PUBLIC_SUPABASE_URL = "https://your-project.supabase.co"
```

Secrets (like `SUPABASE_SERVICE_ROLE_KEY`) must be added via the Cloudflare dashboard under Workers & Pages → Settings → Environment variables.

## Production Checklist

| Item | Check |
|---|---|
| RLS enabled on all tables | Run `SELECT tablename, rowsecurity FROM pg_tables WHERE schemaname = 'public'` — all should be `true` |
| No plaintext secrets in DB | `wallets.vault_secret_id` is a Vault reference, not a raw secret |
| HTTPS only | Enforce via Vercel/Cloudflare — no HTTP in production |
| Service role key server-only | `SUPABASE_SERVICE_ROLE_KEY` never in `NEXT_PUBLIC_*` vars |
| Webhook secret set | `ETHERFUSE_WEBHOOK_SECRET` is set and non-empty |
| Rate limiting active | Upstash Redis connected and tested |
| Stellar mainnet configured | `STELLAR_NETWORK=mainnet` in production env |
| Friendbot disabled | API returns `400` on testnet-only endpoints in prod |
| Auth redirect URL allowlisted | Production domain added to Supabase Auth → URL Configuration |
| Error monitoring | Sentry or similar connected to catch Stellar submission failures |
